# March Madness 2026 -- Bracket Prediction Engine

EMBA 693R Advanced Machine Learning Final Project

Predict every game of the 2026 NCAA Tournament using four independent models, then track how each one performs as real results come in week by week.

## Quick Start

### 1. Install dependencies

From the project folder, make sure these packages are available in your Python environment (Anaconda has all of them by default):

```
pandas  numpy  scikit-learn  joblib  matplotlib  seaborn  streamlit  plotly
```

If you plan to use the **Animal Kingdom** or **Vegas Odds (AI fallback)** models, also install:

```
pip install anthropic
```

### 2. Export the trained models (one-time setup)

Open `march_madness.ipynb` and **Run All Cells**. The final two cells (section 2.7) train regression models and save them to `data/models/`. This only needs to happen once -- the `.pkl` files are reused from then on.

### 3. Set your API key (optional)

Two of the four models call the Anthropic (Claude) API. If you want to use them:

```bash
export ANTHROPIC_API_KEY="sk-ant-your-key-here"
```

Or copy `.env.example` to `.env` and fill in the key. Results are **cached locally** after the first run, so the API is only called once per unique matchup.

### 4. Run the bracket engine

**Option A — Interactive Dashboard (recommended for presentations):**

```bash
pip install streamlit plotly
streamlit run dashboard.py
```

This opens a visual dashboard at http://localhost:8501 with:

- Full tournament bracket with toggle between models
- Champion predictions from every model side by side
- Round-by-round matchup details, upset tracker, path to title
- Accuracy metrics (once actuals are uploaded)
- Model comparison showing where models agree or disagree

**Option B — Jupyter Notebook:**

Open `bracket_engine.ipynb` and **Run All Cells**. You will see:

- Each model's predicted champion
- Round-by-round matchup tables
- A side-by-side comparison of which models agree or disagree

## How It All Fits Together

The project is organized into four layers. Think of it like a pipeline: data flows in at the top, predictions come out at the bottom.

```
┌─────────────────────────────────────────────┐
│  Layer 1 -- SOURCE DATA                     │
│  Team database + bracket structure          │
│  (season_2026.csv, Kaggle seeds & slots)    │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  Layer 2 -- MODELS (x4)                     │
│  Each takes two teams → predicts scores     │
│  Advanced Metrics | Animal Kingdom          │
│  Seeding Only     | Vegas Odds              │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  Layer 3 -- EVALUATION                      │
│  Inject real results, compare accuracy      │
│  across models, round by round              │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  Layer 4 -- FUN ZONE                        │
│  Full bracket printouts, champion paths,    │
│  side-by-side model comparison tables       │
└─────────────────────────────────────────────┘
```

### Layer 1 -- Source Data

All team facts live in a single database object (`TeamDB`). When you call `db.get_team(team_id)`, it returns a dictionary with that team's offensive/defensive ratings, coach stats, win-loss record, and more.

The bracket structure mirrors the NCAA tournament: 4 play-in games, 32 first-round games, then halving each round down to the championship. It is built from the same Kaggle data format used by the March Machine Learning Mania competition.

### Layer 2 -- The Four Models

Every model implements the exact same interface. You give it two teams and it gives back a predicted score for each. This means the bracket engine does not care which model is running -- it just calls `predict()` and records the result.

| Model | What it does | Needs |
|---|---|---|
| **Advanced Metrics** | Uses machine-learned regression on team efficiency stats (adjusted offensive/defensive ratings, coach tournament experience, strength of schedule, etc.). Trained on 15 years of historical tournament data. | `data/models/*.pkl` (generated by running `march_madness.ipynb`) |
| **Animal Kingdom** | Asks Claude AI "if these two mascots fought, who would win?" and uses the fight scores (50-100) as predictions. Pure entertainment value. | `ANTHROPIC_API_KEY` env var |
| **Vegas Odds** | Uses real sportsbook betting lines (spread + over/under) to derive predicted scores. When real lines are not available for a matchup, Claude estimates what Vegas would set the line at -- and those predictions are **flagged as AI-estimated** in the output. | `data/vegas_lines.csv` for real lines, `ANTHROPIC_API_KEY` for AI fallback |
| **Seeding Only** | The simplest model: the higher-seeded team always wins. Ties broken by regular-season win percentage. Scores based on historical averages for each seed matchup. | Nothing (always works) |

### Layer 3 -- Evaluation

Once real tournament results exist, you feed them in as a CSV and the engine compares what each model predicted against what actually happened. The output is an **accuracy matrix** showing:

- Per-round accuracy (e.g., "how did each model do in just the Sweet 16?")
- Cumulative accuracy (e.g., "through the first three rounds combined")
- Spread accuracy (how close were predicted margins to actual margins?)

### Layer 4 -- Fun Zone

Full bracket printouts for each model showing every game from the First Four through the Championship, with predicted scores and the winner marked. After actuals are injected, games are tagged as confirmed or predicted.

## Weekly Workflow

The engine is designed to be re-run each week as real results come in.

**Week 1 (before tournament):** Run `bracket_engine.ipynb` to generate full pre-tournament predictions for all four models.

**Week 2 (Round of 64 complete):**
1. Create `data/actuals.csv` with the real results.
2. Open `bracket_engine.ipynb`, set `ACTUALS_PATH = DATA_DIR / "actuals.csv"` in the Layer 3 cell.
3. Re-run from Layer 3 onward. The engine locks in real R64 outcomes, re-predicts R32 through Championship, and shows the accuracy matrix.

**Week 3, 4, ...:** Same pattern. Append new rows to `data/actuals.csv`, re-run.

For the full walkthrough with slot ID reference tables, team ID lookup, and examples for every round, see **[UPDATING_ACTUALS.md](UPDATING_ACTUALS.md)**.

### Actuals CSV format (quick reference)

```csv
round,winner,winner_score,loser_score
R64,Duke,82,55
R64,Connecticut,75,68
R32,Duke,78,65
Sweet 16,Duke,74,61
```

- `round`: human-readable round name (`R64`, `R32`, `Sweet 16`, `Elite 8`, `Final Four`, `Championship`, or `Play-In`)
- `winner`: team name -- any reasonable spelling works (`Duke`, `duke`, `St Johns`, `UConn`, etc.)
- `winner_score` / `loser_score`: final scores (optional but recommended)

### Vegas lines CSV format

To use real Vegas lines instead of AI estimates, create `data/vegas_lines.csv`:

```csv
team_a,team_b,spread,total,source
Duke,Robert Morris,-22.5,142.5,DraftKings
Florida,Lehigh,-19.5,138.0,FanDuel
```

- `team_a` / `team_b`: team names as they appear in the bracket output
- `spread`: from Team A's perspective (negative means A is favored by that many points)
- `total`: the over/under combined score
- `source`: where you got the line (for your records)

A template is provided at `data/vegas_lines.csv.example`.

## Glossary

| Term | Meaning |
|---|---|
| **Seed** | A team's ranking within their bracket region (1 = best, 16 = worst). There are four regions of 16 teams each. |
| **Spread** | How many points a team is expected to win or lose by. Duke -8.5 means Duke is expected to win by 8.5 points. |
| **Over/under (total)** | The combined score both teams are expected to reach. |
| **Barttorvik** | A college basketball analytics site. Provides adjusted efficiency ratings that account for pace and opponent strength. |
| **adj_em** | Adjusted Efficiency Margin -- the difference between a team's offensive and defensive efficiency. Higher = better. The single most predictive stat. |
| **barthag** | A team's probability of beating an average D-I team on a neutral court. |
| **wab** | Wins Above Bubble -- how many more wins a team has than a bubble (borderline tournament) team would against the same schedule. |
| **First Four** | Four play-in games before the main bracket. Winners advance to the Round of 64. |
| **Kaggle team ID** | A numeric identifier for each team (e.g., Duke = 1181). Used internally by the engine for lookups. |
| **pkl file** | A serialized Python object (a trained model saved to disk so it can be reloaded without retraining). |

## FAQ

**Q: I get "No module named 'engine'"**
Make sure you are running the notebook from the `march_madness/` folder. The notebook adds the current directory to the Python path automatically.

**Q: Advanced Metrics says SKIP**
Run `march_madness.ipynb` all the way through first. The last cells export the trained models to `data/models/`.

**Q: Animal Kingdom / Vegas Odds says SKIP**
Set your Anthropic API key: `export ANTHROPIC_API_KEY="sk-ant-..."` in your terminal before launching Jupyter. Or add it to a `.env` file.

**Q: How do I add a new model?**
Create a new file in `engine/models/` that defines a class inheriting from `PredictionModel`. Implement the `predict()` method (takes two team IDs, returns a dict with `team_a_score`, `team_b_score`, `winner_id`, `confidence`). Then import it in `bracket_engine.ipynb` and add it to the `models` dict.

**Q: Vegas Odds says "AI-ESTIMATED" -- is that bad?**
It means real sportsbook lines were not found in `data/vegas_lines.csv` for that matchup, so the model asked Claude to estimate what Vegas would set. The estimates are reasonable but not authoritative. To use real lines, populate the CSV with actual betting data.
